# Initial System Structure

The system follows a microservice-based client-server architecture deployed inside a Kubernetes cluster. The system is divided into independent subsystems for the frontend client, backend services, GenAI processing, and database management. Communication between all services is handled through REST APIs using JSON over HTTP. 

The architecture consists of a React frontend client, multiple Spring Boot backend microservices, a Python-based GenAI microservice, a PostgreSQL database, an external AI service using the Gemini API, and a docker-based containerized deployment. 

# Client Side : React Frontend

The frontend will be implemented using React and serves as the main user interface of the application. The React client allows students to input academic goals, generate personalized roadmaps, manage to-do-lists, and track progress visually. The frontend communicates with the backend services through the API Gatewy using via OpenAPI SDK. Based on backend responses, the interface dynamically updates roadmap states, milestones, and task completion progress.
Before submitting a goal, the user must select between two input modes “Career Goal” or “University Course Goal”. 

Frontend responsibilities: 
- Accepting user goal input
- Displaying roadmap visualizations
- Displaying completion states and notifications 

# Kubernetes Cluster

All backend-elated services are deployed inside a Kubernetes cluster. The cluster contains the API Gateway, Spring Boot backend services, and the GenAI microservice.
The API Gateway acts as the central entry point for all frontend requests and routes requests to the correct backend service.


# Server Side : Spring Boot Microservices 

The backend will be implemented using Spring Boot and divided into multiple microservices inside the Spring Boot subsystem. These services act as the conection layer between the frontend, GenAI microservice, and database.

## API Gateway

The API receives requests from the React frontend and forwards them to the responsible backend services. It handles request and simplifies communication between the client and the backend microservices.

### 1. Roadmap Service: 

The Roadmap Service stores, retrieves, and updates generated roadmaps. It communicates with both the database and the GenAI microservice.

- Responsibilities:
    - Store generated roadmaps
    - Retrieve roadmap data
    - Update milestones and tasks
    - Synchronize roadmap progress

- (Possible) endpoints:
    - POST /roadmaps ->  creates a new roadmap in the system after it is generated by the GenAI service. It stores milestones and tasks in the database.
    - PUT /roadmaps/{id} -> updates an existing roadmap when a user completes a task.
    - GET /roadmaps/{id} -> retrieves a specific roadmap including all milestones and tasks so that it can be displayed in the frontend.

### 2. Course Service: 

The course service manages TUM course information stored in the databse. It searches course titles and descriptions using keywords extracted from the user prompt.

- Responisbilities:
    - Search TUM course data
    - Retrieve course descriptions, titles, and prerequisites
    - Provide relevant course context to the GenAI service

- (Possible) endpoints:
    - GET /courses/search -> searches the database for courses based on keywords extracted from the user’s goal. It matches course titles and descriptions and returns the relevant results to the GenAI service.
    - GET /courses/{idCourse} -> returns detailed information about a specific course, including description, prerequisites and name. 

### 3. User and Progress Service:
The User and Progress service manages user-specific data and tracks progress of the roadmaps.

- Responsibilities:
    - Store user roadmaps
    - Track completed tasks
    - Update milestone completion states
    - Retrieve saved roadmaps

- (Possible) endpoints:
    - GET /user/{idUser}/roadmaps -> retrieves all saved roadmaps belonging to a specific user.
    - PUT /tasks/{idUser}/{idMap}/complete -> marks a specific task as completed and updates the progress state of the corresponding milestone of the specific roadmap.


# GenAI Service: Python Microservice 

The GenAI component will be implemented as a separate Python microservice inside the Kubernetes cluster. The microservice contains two main components: Python API and LangChain RAG Orchestrator. 

The Python API receives requests from the Roadmap and Course services and is responsible for AI-related functionality. 

The LangChain RAG Orchestrator handles retrieval-augmented generation (RAG). It combines the user prompt with relevant course information retrieved from the Course service before sending the context to the external Gemini API. 

The GenAI service receives an additional parameter from the frontend/backend indicating whether the user selected a Career Goal or a University Course Goal. For the "University Course Goal" mode, the system first verifies whether the requested course exists in the course database. If the course cannot be found in the database, the system does not generate a roadmap and instead returns a message to the frontend informing the user that the requested course is not available. 

The generated roadmap is returned as a structured JSON object containing milestones, tasks, course recommendations, and external suggestions. 

- GenAI responsibilities:
    - Extract keywords from user prompt
    - Retrieve relevant TUM courses
    - Generate structured roadmaps
    - Generate milestone-specific tasks
    - Recommended non TUM-related tasks

## External AI Service

The syste integrates an external AI servie using the Gemini API. The Gemini API is accessed through the LangChain RAG Orchestrator inside the GenAI microservice.

- External AI service responsibilities:
    - Generate roadmap content
    - Structure milestones and tasks
    - Prduce personalized recommendations
    - Create contextual academic guidance

# Database Layer: PostgreSQL

The application uses PostgreSQL as the database. 
The database stores application data such as user information, generated roadmaps, milestones, tasks, completion states, user progress, saved roadmaps, and TUM course information. TUM courses will be displayed as a mock dataset. 

The database contains separate schemas/components for User Schema, Course Schema, and Roadmap Schema. The stored course information includes course titles, course descriptions, prerequisites, and additional metadata.

The Course service queries the PostgreSQL database using keywords extracted from the user prompt. The retrieved courses are then forwarded to the GenAI service as context to reduce hallucinations and improve the recommendation quality.

# System Flow:

1. The student enters a goal into the React frontend and selects either "Career Goal" or "University Course Goal".
2. The frontend sends the request through the API Gateway using REST API calls.
3. The API Gateway forwards the request to the Roadmap microservice.
4. The Roadmap microservice sends the user goal to the GenAI microservice.
5. The Python API triggers the LangChain RAG Orchestrator.
6. The GenAI microservice extracts keywords from the user prompt.
7. The GenAI microservice requests relevant courses from the Course microservice.
8. The Course microservice queries the PostgreSQL database and returns matching courses. 
9. If the user selected “University Course Goal” and the requested course does not exist in the database, the system stops the process and returns an error message to the frontend instead of generating a roadmap.
10. The LangChain RAG Orchestrator combines the user prompt and course context.
11. The extended prompt is sent to the Gemini API.
12. The Gemini API generates a structured roadmap containing milestones and tasks. 
13. The generated roadmap is returned to the Roadmap microservice.
14. The Roadmap microservice stores the roadmap in the database.
15. The frontend retrieves and visualizes the roadmap for the student. 
16. User progress updates are handles through the User and Progress microservice.
